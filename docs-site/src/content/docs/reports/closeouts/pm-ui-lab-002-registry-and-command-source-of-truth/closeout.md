# PM-UI-LAB-002 Registry And Command Source Of Truth Closeout

## Scope

`WR-094` completes the bounded `PM-UI-LAB-002` implementation slice:
Runenwerk editor command affordances now route through one app-owned
`EditorCommandCatalog`, and tool-surface metadata now resolves from
registry-owned `ToolSurfaceDefinition` data before legacy compatibility
adapters.

This slice does not implement the app-hosted Editor Lab surface shell, visual
authoring operations, Editor Lab project IO, diff/apply/rollback, the broader
preview evidence matrix, or final public API/example closeout. Those remain
`PM-UI-LAB-003` through `PM-UI-LAB-007`.

## Implementation Summary

- `apps/runenwerk_editor/src/shell/command_catalog/mod.rs` module
  `command_catalog` defines `EditorCommandCatalog`,
  `EditorCommandDescriptor`, `KnownEditorCommand`, catalog availability rules,
  labels, aliases, route targets, toolbar command projection, shell command
  projection, and typed unavailable-command diagnostic codes.
- `apps/runenwerk_editor/src/shell/command_resolution.rs` module
  `command_resolution` now treats `KnownEditorCommand` and
  `active_route_actions_by_target` as catalog-backed adapters instead of an
  independent route-action source table.
- `apps/runenwerk_editor/src/shell/toolbar_adapter.rs` module
  `toolbar_adapter` derives menu labels, enabled state, disabled state, and
  disabled reasons from the command catalog.
- `apps/runenwerk_editor/src/shell/dispatch_shell_command.rs` function
  `dispatch_toolbar_command` keeps app execution in the app shell while
  consuming catalog-backed availability for unavailable toolbar commands.
  Function `unavailable_toolbar_command` emits typed diagnostics such as
  `[ui:editor.command.unavailable.save_as]`.
- `domain/editor/editor_shell/src/composition/build_editor_shell.rs` function
  `legacy_toolbar_action_for_route_slot` is now explicitly a legacy fallback.
  Normal shell-frame route actions are supplied by
  `EditorShellFrameModel::route_actions_by_route_target`.
- `domain/editor/editor_shell/src/tool_suite/definition.rs` type
  `ToolSurfaceDefinition` now carries registry-owned capability set, retention
  class, creation policy, target-profile compatibility, and optional legacy
  compatibility key.
- `domain/editor/editor_shell/src/tool_suite/registry.rs` type
  `ToolSurfaceRegistry` validates surface labels and required observe
  capability so installed surfaces cannot silently omit source-of-truth
  metadata.
- `domain/editor/editor_shell/src/workspace/surface_contract.rs` functions
  `tool_surface_capabilities_from_registry_or_legacy` and
  `tool_surface_retention_class_from_registry_or_legacy` resolve metadata from
  the installed registry before legacy `ToolSurfaceKind` fallback.
- `apps/runenwerk_editor/src/shell/providers/mod.rs` functions
  `mounted_surface_requests_with_registry`, `stable_key_support`, and
  `stable_keys_support` now use stable keys and registry/provider-family
  metadata for normal mounted-surface requests.
- `apps/runenwerk_editor/src/shell/tool_suites/mod.rs` module
  `tool_suites` and `apps/runenwerk_editor/src/material_lab/tool_suite.rs`
  now register capabilities, retention classes, creation policy, and legacy
  compatibility keys in compiled-in tool-suite surface definitions.
- `assets/editor/ui/editor_bindings.ron` no longer carries the stale disabled
  reason for `window_new_window`; the command is available through the catalog
  and opens a logical editor window.

## Acceptance Evidence

- Catalog descriptor uniqueness and alias uniqueness are covered by
  `editor_command_catalog_has_unique_keys_and_aliases`.
- Checked-in toolbar routes, UI route slots, workspace entries, and menu item
  routes resolve through the catalog in
  `checked_in_toolbar_binding_routes_resolve_to_catalog_descriptors`.
- Shell-frame route coverage for catalog route targets is covered by
  `active_route_actions_cover_every_catalog_route_target_without_shell_fallback`.
- Disabled command projection is covered by
  `unavailable_catalog_commands_project_disabled_routes`.
- Catalog unavailable-command diagnostics are covered by
  `unavailable_toolbar_command_emits_catalog_diagnostic` and by the generated
  runtime proof artifact.
- Registry validation rejects invalid surface labels and missing observe
  capabilities in `ToolSurfaceRegistry` tests.
- Registry-first capability and retention adapters are covered in
  `domain/editor/editor_shell/src/workspace/surface_contract.rs`.
- Provider mounting and mounted-surface request enrichment are covered by
  provider tests and by the generated runtime proof artifact.

## Drift Check

The phase completion drift check found the implementation aligned with the
accepted PM-002 design and WR-094 contract:

- Command execution remains app-owned in `apps/runenwerk_editor`.
- Generic `domain/ui/ui_definition` remains behavior-free and does not own
  editor command behavior, provider behavior, project IO, or runtime activation.
- `domain/editor/editor_shell` owns app-neutral shell projection and registry
  contracts without owning concrete app command execution.
- `ToolSurfaceKind` remains a legacy/persistence compatibility vocabulary,
  while normal mounted-surface requests carry stable keys and registry metadata.
- Later PM-UI-LAB milestones remain incomplete and must not be inferred
  complete from this source-of-truth cleanup.
